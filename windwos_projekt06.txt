<!-- 1. ZIEL -->
<section class="content-section">
<div class="container">
<div class="section-header"><div class="section-number">1</div><div><h2>Ziel</h2><div class="section-tagline">Was wird hier aufgebaut und warum?</div></div></div>
<p class="body-text">In diesem Projekt wird eine <strong>skalierbare und isolierte Unternehmensnetzwerk-Architektur</strong> auf Basis von Windows Server 2019 und Windows 11 Clients aufgebaut. Es simuliert ein reales Szenario: Ein Unternehmen mit mehreren Abteilungen benötigt eine zentral verwaltete IT-Infrastruktur — von der automatischen IP-Vergabe bis zum sicheren Dateiserver.</p>
<p class="body-text">Die drei Kernziele des Projekts sind: <strong>Automatisierung der IP-Vergabe</strong> mittels DHCP, <strong>Standardisierung und Domänenintegration</strong> der Endgeräte über Active Directory sowie die <strong>Bereitstellung eines zentralen File Servers</strong> mit Access-Based Enumeration (ABE) — damit jeder Benutzer nur die Ordner sieht, auf die er Zugriff hat.</p>
<p class="body-text">Der gesamte Prozess wird nach dem <strong>Proof-Driven-Prinzip</strong> dokumentiert — jeder Konfigurationsschritt wird durch Screenshots belegt. Das Ergebnis ist eine produktionsnahe Infrastruktur, wie sie in realen Unternehmen eingesetzt wird.</p>
</div>
</section>

<!-- 4. DURCHFÜHRUNG -->
<section class="content-section">
<div class="container">
<div class="section-header"><div class="section-number">4</div><div><h2>Durchführung</h2><div class="section-tagline">Drei Phasen — von DHCP über Domain Join bis zum File Server</div></div></div>

<!-- PHASE 1 -->
<div class="phase-label">Phase 1 — DHCP-Server einrichten</div>
<p class="body-text">Um die manuelle IP-Konfiguration abzulösen und IP-Konflikte im Netzwerk zu vermeiden, wurde ein zentraler <strong>DHCP-Server</strong> auf dem Windows Server 2019 eingerichtet.</p>

<div class="step-list">
<div class="step-item"><div class="step-num">1</div><div class="step-content"><div class="step-title">DHCP-Rolle installieren</div><div class="step-desc">Im Server-Manager wurde über „Rollen und Features hinzufügen" die Rolle <strong>DHCP-Server</strong> installiert. Nach der Installation erscheint im Server-Manager die Benachrichtigung, dass die DHCP-Konfiguration noch abgeschlossen werden muss.</div></div></div>
<div class="step-item"><div class="step-num">2</div><div class="step-content"><div class="step-title">DHCP im Active Directory autorisieren</div><div class="step-desc">Über den DHCP-Konfigurations-Assistenten wurde der Server mit den Anmeldeinformationen <code>GFN_LAB\Administrator</code> im Active Directory autorisiert. Ohne diese Autorisierung darf ein DHCP-Server in einer AD-Domäne keine Adressen vergeben — ein wichtiger Sicherheitsmechanismus.</div></div></div>
<div class="step-item"><div class="step-num">3</div><div class="step-content"><div class="step-title">IPv4-Bereich (Scope) erstellen</div><div class="step-desc">In der DHCP-Konsole wurde ein neuer IPv4-Bereich <strong>„GFN_Lab_Clients"</strong> erstellt: Start-IP <code>192.168.10.51</code>, End-IP <code>192.168.10.100</code>, Subnetzmaske <code>255.255.255.0</code> (/24). Damit stehen 50 dynamische Adressen für Clients zur Verfügung.</div></div></div>
<div class="step-item"><div class="step-num">4</div><div class="step-content"><div class="step-title">Bereichsoptionen konfigurieren</div><div class="step-desc">Als <strong>Standardgateway</strong> (Router) wurde <code>192.168.10.254</code> eingetragen. Als <strong>DNS-Server</strong> wurde <code>192.168.10.10</code> (der Domain Controller) und als <strong>Übergeordnete Domäne</strong> <code>gfn_lab.local</code> angegeben. Damit erhalten alle Clients automatisch die korrekten Routing- und DNS-Informationen.</div></div></div>
</div>

<div class="phase-label">Phase 2 — Client-Standardisierung & Domain Join</div>
<p class="body-text">Alle Windows 11 Clients wurden nach einem <strong>einheitlichen Namensschema</strong> benannt und in die Active-Directory-Domäne aufgenommen. Ziel ist eine zentrale Verwaltbarkeit aller Endgeräte.</p>

<div class="step-list">
<div class="step-item"><div class="step-num">1</div><div class="step-content"><div class="step-title">IPv4 auf DHCP umstellen</div><div class="step-desc">Auf jedem Client wurde über <code>ncpa.cpl</code> → Ethernet → IPv4-Eigenschaften die Option <strong>„IP-Adresse automatisch beziehen"</strong> aktiviert. Der DNS-Server wurde manuell auf <code>192.168.10.10</code> gesetzt, um die Domänenauflösung sicherzustellen.</div></div></div>
<div class="step-item"><div class="step-num">2</div><div class="step-content"><div class="step-title">DHCP-Lease überprüfen</div><div class="step-desc">Über <code>ipconfig /all</code> wurde auf jedem Client bestätigt, dass die IP-Adresse vom DHCP-Server bezogen wurde. Beispiel: Hostname <strong>Win11_Pro</strong>, IP <code>192.168.10.52</code>, DHCP-Server <code>192.168.10.10</code>, DNS-Suffix <code>gfn_lab.local</code>, Lease gültig bis 31.03.2026.</div></div></div>
<div class="step-item"><div class="step-num">3</div><div class="step-content"><div class="step-title">Computerumbenennung (NetBIOS-konform)</div><div class="step-desc">Jeder Client wurde über <code>sysdm.cpl</code> nach einem professionellen Schema umbenannt: <strong>W11P-Finanz1</strong>, <strong>W11P-Finanz2</strong>, <strong>W11P-Vertrieb1</strong>, <strong>W11P-Vertrieb2</strong>, <strong>W11P-Betrieb</strong>, <strong>W11ENT-IT</strong>. Die Namen sind maximal 15 Zeichen lang (NetBIOS-Limit) und zeigen auf den ersten Blick die Abteilung.</div></div></div>
<div class="step-item"><div class="step-num">4</div><div class="step-content"><div class="step-title">Domain Join</div><div class="step-desc">Über <code>sysdm.cpl</code> → „Ändern" → „Domäne" wurde jeder Client der Domäne <strong>gfn_lab.local</strong> hinzugefügt. Nach dem Neustart wird die erfolgreiche Integration in der ADUC-Konsole (Active Directory-Benutzer und -Computer) unter dem Container „Computers" sichtbar.</div></div></div>
</div>

<div class="phase-label">Phase 3 — Zentraler File Server mit ABE</div>
<p class="body-text">Für die abteilungsübergreifende Zusammenarbeit wurde ein <strong>zentraler Dateiserver</strong> eingerichtet, der höchste Sicherheits- und Usability-Standards erfüllt. Jeder Benutzer soll nur die Ordner sehen, auf die er Zugriff hat.</p>

<div class="step-list">
<div class="step-item"><div class="step-num">1</div><div class="step-content"><div class="step-title">Ordnerstruktur erstellen</div><div class="step-desc">Auf dem Server wurde unter <code>C:\Unternehmensdaten</code> eine professionelle Ordnerstruktur mit 5 Abteilungen angelegt: <strong>IT_Abteilung</strong>, <strong>Finanz_Abteilung</strong>, <strong>Vertrieb_Abteilung</strong>, <strong>Betrieb_Abteilung</strong> und <strong>Gemeinsam</strong> (für abteilungsübergreifende Dokumente).</div></div></div>
<div class="step-item"><div class="step-num">2</div><div class="step-content"><div class="step-title">SMB-Freigabe konfigurieren</div><div class="step-desc">Der Stammordner wurde als Netzwerkfreigabe über „Erweiterte Freigabe" konfiguriert. Der Freigabename lautet <strong>Unternehmensdaten</strong>, der UNC-Pfad ist <code>\\WIN-QLN9UGPDOEA\Unternehmensdaten</code>. Auf der Freigabe-Ebene wurde die Gruppe „Jeder" mit Vollzugriff berechtigt — die granulare Steuerung erfolgt über NTFS-Berechtigungen.</div></div></div>
<div class="step-item"><div class="step-num">3</div><div class="step-content"><div class="step-title">Access-Based Enumeration (ABE) aktivieren</div><div class="step-desc">Im Server-Manager unter <strong>Datei- und Speicherdienste → Freigaben</strong> wurde die Funktion „Zugriffsbasierte Aufzählung aktivieren" eingeschaltet. Dadurch werden Benutzern nur die Ordner angezeigt, für die sie tatsächlich Lese- oder Schreibrechte besitzen — ein unverzichtbares Sicherheitsfeature in Unternehmensumgebungen.</div></div></div>
</div>

<!-- 5. PROBLEME & LÖSUNG -->
<section class="content-section">
<div class="container">
<div class="section-header"><div class="section-number">5</div><div><h2>Probleme & Lösungen</h2><div class="section-tagline">Herausforderungen im Prozess und wie sie gelöst wurden</div></div></div>

<div class="issue-card">
<div class="issue-header problem"><svg viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2"><circle cx="12" cy="12" r="10"/><line x1="12" y1="8" x2="12" y2="12"/><line x1="12" y1="16" x2="12.01" y2="16"/></svg>Problem 1: Falsches Standardgateway in den Bereichsoptionen</div>
<div class="issue-body">Nach der Erstellung des DHCP-Scopes wurde festgestellt, dass in den <strong>Bereichsoptionen unter „003 Router"</strong> eine falsche IP-Adresse eingetragen war: <code>10.100.15.41</code> — eine Adresse aus dem externen Lab-Netzwerk des Remote Desktop Servers, nicht aus dem internen 192.168.10.x-Netz. Clients, die diese Option erhielten, konnten keine Verbindung zum Internet aufbauen.</div>
</div>

<div class="issue-card">
<div class="issue-header solution"><svg viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2"><path d="M22 11.08V12a10 10 0 1 1-5.93-9.14"/><polyline points="22 4 12 14.01 9 11.01"/></svg>Lösung: Gateway-Adresse manuell korrigieren</div>
<div class="issue-body">In der DHCP-Konsole unter <strong>Bereichsoptionen → 003 Router</strong> wurde die falsche Adresse <code>10.100.15.41</code> entfernt und durch die korrekte Gateway-Adresse <code>192.168.10.254</code> ersetzt. Anschließend wurde auf den betroffenen Clients <code>ipconfig /release</code> und <code>ipconfig /renew</code> ausgeführt, um die korrigierte Konfiguration sofort zu übernehmen.<br><br><strong style="color:var(--win-blue)">Praxishinweis:</strong> In virtualisierten Umgebungen existieren häufig mehrere Netzwerkbereiche parallel. Es ist entscheidend, die Bereichsoptionen nach der Erstellung zu überprüfen — insbesondere den Router-Eintrag. Ein falsches Gateway führt dazu, dass Clients zwar intern kommunizieren können, aber keinen Zugang zum Internet oder zu anderen Subnetzen erhalten.</div>
</div>

<figure class="proof-figure" onclick="openLB(this)"><img src="images/Projekt6_14.png" alt="Bereichsoptionen mit falschem Router" loading="lazy"><figcaption class="proof-caption"><span class="proof-num">Abb. 15</span><span class="proof-text">Bereichsoptionen — Sichtbar: 003 Router mit der falschen externen IP 10.100.15.41 (markiert). DNS und Domäne korrekt.</span></figcaption></figure>

<div class="issue-card" style="margin-top:32px">
<div class="issue-header problem"><svg viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2"><circle cx="12" cy="12" r="10"/><line x1="12" y1="8" x2="12" y2="12"/><line x1="12" y1="16" x2="12.01" y2="16"/></svg>Problem 2: Domain Join schlägt fehl — DNS-Auflösung</div>
<div class="issue-body">Beim Versuch, einen Client der Domäne <code>gfn_lab.local</code> hinzuzufügen, erschien die Fehlermeldung: <strong>„Die Domäne konnte nicht kontaktiert werden"</strong>. Der Client konnte den Domänennamen nicht auflösen, weil er noch den externen DNS-Server (z.&nbsp;B. den des ISP) als bevorzugten DNS-Server eingetragen hatte.</div>
</div>

<div class="issue-card">
<div class="issue-header solution"><svg viewBox="0 0 24 24" fill="none" stroke="currentColor" stroke-width="2"><path d="M22 11.08V12a10 10 0 1 1-5.93-9.14"/><polyline points="22 4 12 14.01 9 11.01"/></svg>Lösung: DNS-Server manuell auf den DC setzen</div>
<div class="issue-body">In den IPv4-Eigenschaften des Clients wurde der <strong>bevorzugte DNS-Server</strong> manuell auf <code>192.168.10.10</code> (den Domain Controller) gesetzt. Alternativ kann die IP auf DHCP umgestellt werden — vorausgesetzt, die DHCP-Bereichsoptionen enthalten den korrekten DNS-Eintrag. Nach der Änderung war der Domain Join sofort erfolgreich.<br><br><strong style="color:var(--win-blue)">Praxishinweis:</strong> In einer Active-Directory-Umgebung muss der Client den Domain Controller als DNS-Server verwenden, da AD-Dienste (SRV-Records) nur über den internen DNS aufgelöst werden können. Dies ist eine der häufigsten Fehlerquellen beim Domain Join.</div>
</div>

<p class="body-text" style="margin-top:32px"><strong>Abweichungen vom Standardvorgehen:</strong></p>
<div class="note-block"><div class="note-block-title">Client-Benennung</div><div class="note-block-text">Statt generischer Namen wie „PC-01", „PC-02" wurden abteilungsbezogene Namen gewählt (W11P-Finanz1, W11ENT-IT etc.). Dies entspricht dem NetBIOS-Limit von 15 Zeichen und ist praxisnäher für reale Unternehmensumgebungen.</div></div>
<div class="note-block"><div class="note-block-title">Freigabe-Berechtigungen</div><div class="note-block-text">Auf der Freigabe-Ebene wurde „Jeder = Vollzugriff" gesetzt. Die granulare Zugriffskontrolle erfolgt ausschließlich über NTFS-Berechtigungen. Dieses Vorgehen entspricht der Microsoft Best Practice: Freigabe weit öffnen, NTFS steuert.</div></div>
</div>

<p>Dieses Projekt zeigt den Aufbau einer <strong>vollständigen Unternehmensinfrastruktur</strong> in drei Schritten: Netzwerkautomatisierung (DHCP), zentrales Client-Management (AD Domain Join) und sichere Dateiablage (File Server mit ABE).
